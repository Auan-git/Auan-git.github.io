第四次讲座

第 3 次讲座：详细内容

军训第 3 周 · 90 分钟

主题：老师不教但你必须会的

基调：快节奏、高密度、大量现场演示，像一份”生存技能速成课”

讲前准备清单

准备事项 说明
电脑 + 投影 这次有大量现场演示，确保投影清晰、字体调大
提前测试演示环境 确保 Git、Python、VS Code、Terminal 在能正常运行，投影时不出幺蛾子
备用方案 提前把需要的包下载好，能离线演示的尽量离线
Markdown 速查卡 Markdown 语法速查表，现场展示 + 讲完后拍照发群

开场（5 分钟）

“前三周我们聊了 CS 是什么、课表怎么安排的。今天不一样——今天不聊大道理，聊实打实的技能。”

“你们接下来四年会遇到很多东西，老师不会教、课表里没有、但你迟早要用到。有些人大一就学会了，效率碾压同龄人；有些人大三才零零碎碎接触到，浪费了两年时间。”

“今天我要做的就是把这些东西打包一次性给你们。内容会比较密，节奏会比较快，有些东西你今天听不懂没关系

（看举手情况）

“好的，带了的同学待会可以跟着我一起操作。没带的也没关系，看我演示就行，回去自己试。”

第一块：学习方式的转变（10 分钟）

“在讲具体工具之前，我先说一件更重要的事——学习方式怎么变。”

“你们从小学到高中，学了 12 年。这 12 年里，学习模式是固定的：老师上课讲 → 你记笔记 → 回家做题 → 考试。有人追着你学，有明确的进度，有标准答案。”

“大学完全不同。我把区别列出来你们感受一下。”

（PPT 切到对比表）

高中模式 大学模式
谁来安排学什么 老师/学校 你自己
知识从哪来 课本 + 老师讲 官方文档 + 教材 + 视频 + 论文 + 开源代码
怎么解决问题 问老师 先搜索 → 再看文档 → 再问同学 → 最后问老师
怎么验证学没学会 考试分数 能不能做出来（写代码/做项目/解决问题）
学习节奏 全班统一 每个人不同，自己把握
最大的敌人 不够努力 没有方向 + 信息过载 + 自律不足

“你以后会发现，大学里真正拉开差距的不是智商，是自学能力。同样的课，有人只学了课本上的，有人已经在 GitHub 上做了三个项目。差别不在于谁更聪明，而在于谁更会自己找资源、自己学。”

“自学能力是什么？我总结成四件事：”

  1. 搜索能力 — 知道去哪找答案

  2. 阅读能力 — 能看懂英文文档和技术文章

  3. 实践能力 — 看完就动手，不是只收藏

  4. 坚持 — 一个东西学不懂就多看几遍，不要放弃

“这四件事，今天后面的内容全部围绕它们展开。”

第二块：搜索、读报错、提问（15 分钟）

“先讲搜索。这听起来很基础，但信不信——大部分大一新生不会搜索。”

一、怎么搜索

“你们平时遇到问题怎么搜？百度一下？”

“我告诉你们一个事实：在技术领域，Google 和edge的质量远高于百度。 不是因为百度不好，而是因为全世界的技术内容、技术问答主要在英文互联网上。Stack Overflow、GitHub、官方文档、技术博客——这些最优质的资源基本都是英文的。”

（PPT 切到搜索对比）

搜索方式 效果
百度 “Python 列表怎么排序” 大量 CSDN 复制粘贴内容，质量参差不齐，可能过时广告
Google “python sort list” Stack Overflow 高赞回答，官方文档，准确且最新
百度 “TypeError cannot read property” 一堆答非所问的结果
Google “TypeError cannot read property of undefined javascript site:stackoverflow.com” 精准命中，第一条就是答案

“几个搜索技巧：”

  1. 用英文搜。”python list sort” 比 “python 列表排序” 结果好得多。你的英语不需要很好，搜几个关键词就够了。

  2. 直接搜报错信息。程序报错了一长串？把关键那行复制下来，去掉你自己的文件路径，直接搜。

  3. 加限定词。在搜索词后面加 site:stackoverflow.com 就只搜这个网站。

  4. 搜官方文档。比如 “flask quickstart” 而不是 “flask 怎么用”。

“我给你们一个搜索优先级，从高到低：”

“以后遇到问题，按这个顺序来，效率能提高好几倍。”

二、怎么读报错信息

“大一新生遇到报错最常见的反应：截图 → 发群里 → 问’这是怎么回事’。”

“我先不说这种方式好不好，我先教你们一个更好的方法——自己读懂报错信息。”

（打开一个终端，现场故意写一段有 bug 的代码）

运行，展示报错：

“看着很长很吓人？其实只需要看三个东西：”

  1. 哪一行出了错？ → File "test.py", line 2 — 第 2 行

  2. 什么类型的错误？ → IndexError — 索引错误

  3. 具体什么问题？ → list index out of range — 列表越界了

“然后你就明白了：numbers 只有 3 个元素（索引 0、1、2），你要取索引 5，不存在，所以报错。”

“再看一个例子：”

报错：

“错误类型是 AttributeError，意思是’字符串没有叫 length 的属性’。为什么？因为 Python 里获取字符串长度不是 .length，而是 len(name)。”

“学会读报错信息，你就解决了 80% 的编程问题。 报错信息不是给你看的天书，它是编译器/解释器在告诉你哪里出了问题。养成读报错信息的习惯，而不是一看到红字就慌。”

三、怎么提问

“有些问题你确实搜不到、看不懂，需要问人。怎么问？”

“你们以后会加很多群——课程群、技术群、社团群。在群里提问是有讲究的。”

（PPT 展示对比）

❌ 错误提问：

“我的代码跑不出来了，帮我看看”

问题在哪： 什么代码？什么报错？什么环境？试过什么？别人想帮你也帮不了。

❌ 错误提问：

“Python 怎么用？”

问题在哪： 这个问题大到能写一本书，没人有时间回答。

✅ 正确提问：

“我在用 Python 读取一个 CSV 文件，第 15 行报了 UnicodeDecodeError: 'utf-8' codec can't decode byte 0xb5 in position 12。我试过加 encoding='latin-1' 可以解决，但我想知道为什么会这样。环境是 Python 3.11，Windows 11。”

好在哪： 说了做什么、报了什么错、试过什么、想知道什么、环境是什么。别人一看就知道怎么帮你。

“总结一下提问的公式：”

① 你在做什么 → ② 你遇到了什么问题（贴报错信息）→ ③ 你试过什么 → ④ 你的环境是什么

“做到这四点，你问出的问题质量就超过了 90% 的人。而且你会发现——很多时候你在组织问题的过程中，自己就想明白了。”

“有一篇文章叫《提问的智慧》（How To Ask Questions The Smart Way），强烈建议你们去读一下，网上有中文翻译。”

第三块：写作工具（15 分钟）

一、Markdown（现场演示，10 分钟）

“接下来讲一个你们可能没听说过、但以后会天天用的东西——Markdown。”

“先问一下：有多少人听说过 Markdown？”

（看举手）

“不多，正常。但我要告诉你们——你们以后写技术笔记、写博客、写 GitHub 上的项目说明、写课程实验报告，大概率都会用 Markdown。它比 Word 轻量十倍，比纯文本好看十倍。”

“Markdown 是什么？一句话：一种用纯文本写带格式文档的语法。 你不需要用鼠标点按钮加粗、调字号，只需要在文字前后加几个符号。”

“我现场演示一下。”

（打开 VS Code，新建一个 .md 文件，边写边讲）

表格

语言 用途
Python AI、脚本、Web
C 底层、操作系统

链接和图片

GitHub

这是一段引用文字

写完之后，用 VS Code 的预览功能看一下效果。

（现场切到 VS Code 的 Markdown 预览模式，展示渲染后的效果）

“看到了吗？几行简单的文本符号，渲染出来就像一篇排版精美的文档。而且它是纯文本——很小、不会乱码、任何编辑器都能打开、Git 可以追踪修改。”

“你们以后会在这些地方用到 Markdown：”

“一个建议：从今天开始，你的所有笔记都用 Markdown 写。 用 VS Code 打开 .md 文件就行，按 Ctrl+Shift+V 可以预览。写多了你就上瘾了——因为比 Word 快太多了。”

“Markdown 语法 30 分钟就能学完，我这里只展示了最常用的。完整的语法参考，我会放在《生存手册》里。”

二、LaTeX + 技术英语（快速过，5 分钟）

“Markdown 适合日常写作。但如果你要写数学公式很多的论文或报告，你需要 LaTeX。”

“LaTeX 是学术界的标准排版工具。你以后考研读研写论文、或者课程大作业要求写报告，大概率要用它。”

“我只告诉你们三件事：”

  1. 它是什么： 一种排版系统，用纯文本写，编译后生成精美的 PDF

  2. 什么时候用： 写论文、写数学公式多的报告、写毕业论文

  3. 去哪学： Overleaf（overleaf.com）— 一个在线 LaTeX 编辑器，免费，有大量模板。你可以在上面找一个模板，改一改就能用。不要从零学起，套模板就行。

“现在不需要深入学，知道有这个东西就行。等你大三大四写论文的时候再去学，完全来得及。”

“最后说一下技术英语。”

“我不打算教你们怎么学英语，但想说一件事：你们以后读的最好的技术资料，都是英文的。 官方文档是英文、Stack Overflow 是英文、最新的论文是英文、最好的教程是英文。”

“很多同学习惯找中文翻译版，但翻译版通常有两个问题：一是不全，二是过时。直接读原文，信息损耗最少。”

“你们不需要’英语很好’才能读技术文档。技术文档的词汇量很小，常用词就那几百个。Chrome 装个翻译插件，遇到不认识的词查一下，读多了就习惯了。”

“一个建议：从现在开始，每周读一篇英文技术文章。哪怕只有 500 字，坚持一个学期，你的技术英语阅读能力就不一样了。“

第四块：开发工具（25 分钟）

一、命令行（现场演示，8 分钟）

“接下来讲命令行。我知道你们很多人看到黑框框会害怕，觉得这是’黑客用的东西’。但命令行不是黑客专属——它是开发者每天的日常。”

“先说为什么你们要学会用命令行。”

  1. 服务器只有命令行。 你以后连接远程服务器（部署网站、跑代码、做实验），没有图形界面给你用。

  2. 很多工具只有命令行版本。 Git、Docker、包管理器……很多操作只能在命令行完成。

  3. 效率高。 用鼠标点来点去的操作，命令行一行搞定。

  4. 这是专业和业余的分界线。 会用命令行的人看不会用的人，就像老司机看刚上路的新手。

“我现场给你们演示一些基本操作。不用记，知道有这些命令就行。”

（打开终端，现场演示）

“这几个命令——ls、cd、mkdir、cat、cp、rm、grep——是最基础的，你以后每天都会用到。不用背，用到的时候查一下就行，用多了自然就记住了。”

“再演示一个管道操作，这是命令行最强大的特性之一：”

“| 叫做管道，意思是’把左边的输出当作右边的输入’。你可以像搭积木一样把多个命令串起来，处理非常复杂的任务。这就是命令行的威力。”

“还有一个重要概念：SSH。SSH 就是远程连接另一台电脑的工具。你们以后会经常用到。”

“现在知道有这么个东西就行。等你们以后需要用服务器的时候，我再教你们具体操作。”

二、Git 进阶（现场演示，10 分钟）

“上周让你们注册了 GitHub，今天教你们怎么真正用起来。”

“先说 Git 和 GitHub 的区别：”

  • Git 是一个工具，装在你电脑上，管理代码的版本

  • GitHub 是一个网站，把你的代码放到云端，别人能看到、能协作

“打个比方：Git 是你的本地笔记本，GitHub 是你把笔记发到网上的博客。”

“我演示一个完整的流程：从创建仓库到推代码到 GitHub。“

（现场演示，边做边讲）

第一步：在 GitHub 上创建仓库

“打开 github.com，点右上角的 ‘+‘，’New repository’。名字叫 my-first-repo，勾上 ‘Add a README file’，创建。”

第二步：克隆到本地

第三步：创建一个文件并提交

第四步：打开 GitHub 看效果

“现在打开你的 GitHub 仓库页面，刷新——看到了吗？你的代码已经在上面了。”

“这就是 Git 的基本工作流程：改代码 → add → commit → push。你以后每天的工作就是这个循环。”

“再补充几个你们以后会用到的 Git 概念：”

“分支（branch） 是 Git 最强大的功能之一。打个比方：你写代码写到一半，突然想试一个新想法，但又怕把原来的搞坏。你可以创建一个’分支’，在分支上随便试，搞砸了就删掉，原来的代码不受影响。”

“还有一个重要概念——Pull Request（PR）。以后你们在公司或开源项目里协作，不是直接往主代码里提交，而是先提交一个 PR，让别人审查你的代码，通过了才合并进去。这就是’代码审查’（Code Review）。”

“Git 的概念很多，今天只建立一个基本印象。建议你们回去自己走一遍刚才的流程——创建仓库、克隆、改代码、提交、推送。做一遍就记住了。”

三、VS Code 快速介绍（3 分钟）

“说到写代码，推荐你们用 VS Code。免费、轻量、插件多，是目前全世界最流行的编辑器。”

“我快速演示几个你们马上就能用到的功能：”

（打开 VS Code，快速演示）

  1. 安装中文语言包 — 扩展商店搜 “Chinese”，装上，重启就是中文界面

  2. 安装 Python 扩展 — 搜 “Python”，装上，就能写 Python 了

  3. 内置终端 — 按 `Ctrl+`` 打开终端，不用切换窗口

  4. Markdown 预览 — 打开 .md 文件，按 Ctrl+Shift+V 预览

  5. Git 集成 — 左侧有个源代码管理图标，改了哪些文件一目了然

  6. 快捷键 — Ctrl+P 快速打开文件，Ctrl+Shift+F 全局搜索

“VS Code 的功能很多，但你们不需要一次学完。先装上、能写代码就行，用到哪个功能再学哪个。”

第五块：数据格式与实用技能（10 分钟）

一、JSON（5 分钟）

“接下来讲两个你们以后会天天打交道的数据格式——JSON 和 YAML。”

“先说 JSON。JSON 是什么？程序之间互相传递数据的语言。 你们以后调 API（就是从网上获取数据）、写配置文件、前后端数据交换——几乎全是 JSON。”

“它的格式非常简单，看一下就懂：”

“就这么简单。花括号 {} 表示一个对象，方括号 [] 表示一个列表，键值对用冒号分隔。数据类型就四种：字符串（带引号）、数字、布尔值（true/false）、null。”

“你以后读 API 文档，返回的数据格式就是 JSON。你会看到这样的东西：”

“JSON 看 10 分钟就能懂。但你得先知道它存在，否则第一次遇到会懵。”

二、YAML（2 分钟）

“YAML 和 JSON 很像，但更人类友好。你们以后配 Docker、配 CI/CD、配各种开发工具，配置文件几乎都是 YAML。”

“看到了吗？没有花括号、没有引号（大多数情况），用缩进表示层级。读起来比 JSON 舒服。”

“JSON 和 YAML 都不需要专门学，10 分钟看一下格式就行。但你要先知道它们是什么，否则以后遇到配置文件会一脸懵。”

三、正则表达式（3 分钟）

“最后一个：正则表达式，英文叫 Regex。这东西长得像天书，但它是文本处理的瑞士军刀。”

“什么意思？假设你有一大堆文本，你想找出里面所有的邮箱地址。一个一个肉眼找？不可能。写个程序一个个字符判断？太麻烦。用正则表达式？一行搞定：”

“看着像乱码？但每个符号都有含义。我拆开看：”

“正则表达式你在这些地方会用到：数据清洗（从一堆日志里提取信息）、表单验证（检查邮箱格式对不对）、搜索替换（批量修改代码里的变量名）。”

“不需要精通，但要知道有这个东西，遇到文本处理问题能想到’哦，这个正则可以搞定’。推荐一个在线练习网站：regex101.com，把你的正则贴上去，实时看到匹配结果，非常好用。”

第六块：包管理器与虚拟环境（5 分钟）

“你们以后写代码，不可能什么都自己从零写。别人已经写好了很多好用的库（library），你直接拿来用就行。”

“怎么安装这些库？——通过包管理器。”

“以 Python 为例：”

“Node.js（JavaScript）的包管理器叫 npm，Java 的叫 Maven 或 Gradle。原理一样——你告诉包管理器’我要用这个库’，它帮你下载、安装、管理。”

“然后是一个重要概念——虚拟环境。”

“为什么需要虚拟环境？假设你有两个项目：项目 A 需要 numpy 1.20，项目 B 需要 numpy 1.24。如果你只用一个全局环境，装了 1.24，项目 A 可能就跑不了了。”

“虚拟环境就是给每个项目创建一个独立的’空间’，每个空间装自己需要的库，互不影响。”

“一个建议：每个项目都用虚拟环境。 这是专业开发者的习惯，越早养成越好。”

第七块：数字素养与安全（5 分钟）

“最后讲几个安全相关的习惯。不是什么高深的技术，但能帮你避免很多麻烦。”

一、密码管理

“你们的密码是不是’一个密码走天下’？所有网站都用同一个密码？”

“这非常危险。一旦某个网站被泄露（这种事经常发生），你所有的账号都完了。”

“建议：用密码管理器。 推荐 Bitwarden（免费开源）。你只需要记住一个主密码，其他所有密码都由它生成和管理。每个网站用不同的随机密码，泄露一个也不影响其他的。”

二、两步验证（2FA）

“GitHub、邮箱这些重要账号，一定要开两步验证。开了之后，就算别人知道你的密码，没有你的手机验证码也登不进去。”

“GitHub 的两步验证路径：Settings → Password and authentication → Two-factor authentication。今天回去就开。”

三、不要把密码提交到 Git

“这是新手最容易犯的错之一。你写代码的时候可能需要填 API 密钥、数据库密码。如果你直接写在代码里，然后 git push——恭喜，你的密码现在全世界都能看到了。”

“正确做法：把密码放在环境变量或 .env 文件里，并且把 .env 文件加到 .gitignore 里（告诉 Git 不要追踪这个文件）。”

“这些细节以后具体学，现在只需要记住：密码和密钥永远不要出现在你的代码仓库里。“

四、不要随便运行网上的代码

“你们以后会在网上复制很多代码——Stack Overflow 上、GitHub 上、博客上。复制之前，至少看懂它在做什么。”

“尤其是那些需要管理员权限的命令（sudo 开头的），或者需要你关闭安全软件的——一定要搞清楚它到底在干什么再运行。网上有人故意分享有恶意的代码，新手很容易中招。”

第八块：开源精神

“刚才讲了五条应对策略。在讲具体的 AI 工具之前，我想插入一个话题——开源。”

“为什么在讲 AI 的时候突然讲开源？因为你们现在用的几乎所有 AI——DeepSeek、Llama、Stable Diffusion、Hugging Face 上几千个模型——全部是开源的。”

“而且不只是 AI。你们以后学的 Linux、用的 Git、写代码的 VS Code、跑模型的 PyTorch——全是开源的。“

“如果你们不了解开源，你们就不了解计算机世界真正的运转方式。”

一、什么是开源（2 分钟）

“先说清楚’开源’到底是什么。”

“很多人以为开源就是’免费用’。不是。开源的核心不是免费，是开放。“

“什么意思？——源代码公开，任何人都可以查看、使用、修改、分发。“

“举个例子。你们用的 VS Code，微软把它的每一行源代码都公开放在 GitHub 上。你可以去看它是怎么写的，可以自己修改功能，甚至可以基于它做一个自己的编辑器出来。”

“对比一下：你用微信，但你永远看不到微信的源代码。你不知道它底下怎么运行的，你不能修改它，你不能基于它做自己的产品。这是闭源。”

“再看 Linux：Linus Torvalds 把整个操作系统的源代码公开了。全世界任何人都可以看、可以用、可以改。这就是开源。”

“两种模式没有绝对的对错，但在 CS 领域——尤其是学习阶段——开源对你们的价值是巨大的。“

二、为什么 CS 学生必须了解开源（3 分钟）

原因一：你用的一切都建立在开源之上

“你们以后四年会用到的东西，我列一下：”

你用的 它是开源的
Linux 开源操作系统，全世界 90%+ 服务器跑的都是它
Git 开源版本管理工具，Linus Torvalds 写的
VS Code 开源编辑器，微软开源的
Python 开源编程语言
PyTorch / TensorFlow 开源深度学习框架
MySQL / PostgreSQL 开源数据库
Docker 开源容器化工具
LLaMA / DeepSeek 开源大语言模型

“你们以后写的每一行代码、跑的每一个模型、用的每一个工具，几乎都站在开源的肩膀上。不理解开源，就像住在一个城市里却不知道这个城市的水电是从哪来的。“

原因二：读开源代码是最好的学习方式

“你们想提高编程能力？最快的路径之一就是读别人的代码。”

“读什么代码？——读开源项目。全世界最优秀的工程师写的代码，每一行都公开在那里，随便你看。”

“你学数据结构，可以去看 Redis 怎么实现的跳表。你学操作系统，可以去看 Linux 内核的进程调度。你学 AI，可以去看 Hugging Face 的 Transformers 库是怎么组织的。”

“这些代码比任何教科书都有价值——因为它们是真正生产环境中跑着的代码，经过了千万人的审查和优化。”

原因三：你的 GitHub 就是你的简历

“我之前在第 3 次讲座提过 GitHub，今天再强调一次。”

“以后你们找实习、找工作、考研面试，面试官大概率会看你的 GitHub 主页。”

“他看什么？——看你参与了什么开源项目、你自己的项目有多少 star、你的代码质量怎么样、你给社区贡献了什么。“

“一份 GitHub 主页，比简历上写的’精通 Java’有说服力一万倍。因为代码不会撒谎。”

原因四：AI 时代开源比以往更重要

“回到 AI 话题。2026 年一个很重要的趋势是——开源 AI 正在快速追赶甚至超越闭源 AI。“

“Meta 开源了 LLaMA 系列，DeepSeek 完全开源，Hugging Face 上有几万个开源模型。MCP 协议让所有工具可以互通——它也是开放标准。”

“为什么这些大公司要开源？因为开源能聚集全世界的智慧。一个人写不出 Linux，但全世界几万个开发者一起写，就写出了世界上最重要的操作系统。”

“AI 时代，开源意味着：你可以站在全世界最聪明的人的肩膀上，而不是从零开始。“

三、开源的底层精神（2 分钟）

“开源不只是一个’商业模式’或者’技术策略’。它背后有一套精神——我觉得这套精神和计算机科学本身非常契合。”

“《理工科计算机基础指北》里有一段话我特别喜欢——“

“‘在理工科世界，开源是协作范式、方法论甚至精神信仰。’“

“它包含几个核心理念：”

“① 开放与透明。 代码白盒，所有人都能审查。你做得好不好，不是靠说的，是靠代码说话。这就像科学论文要公开实验过程，让别人能复现。”

“② 协作与共享。 你解决了一个问题，把方案分享出来，全世界的人都不用再踩同一个坑。你用了别人的方案，在此基础上做出更好的东西，再分享出去。知识在流动中变得更好。”

“③ 精英治理。 在开源社区里，你的发言权不取决于你的职位、你的学历、你的年龄，而取决于你代码的质量。一个大一学生提交的代码如果比教授的好，社区会接受学生的。”

“④ 解决问题导向。 开源项目的诞生几乎都是因为一个实际问题：Linux 是因为 Linus 想在自己的电脑上跑一个操作系统，Git 是因为 Linux 内核的版本管理工具不够用了。最好的项目不是想出来的，是做出来的。“

四、CS 学生如何参与开源（2 分钟）

“你们可能觉得’开源好高大上，我现在什么都不会，怎么参与？’”

“其实入门的门槛比你们想象的低得多。参与开源不只是写代码。”

方式一：从使用开始

“最简单的参与方式——用它。 你用了 VS Code、用了 PyTorch、用了某个开源工具，在使用过程中发现了 bug 或者觉得文档不清楚——去 GitHub 上提一个 Issue。”

“一个高质量的 Issue 本身就是对社区的贡献。”

方式二：改文档、修错别字

“你们别笑——很多开源大佬的第一步就是修错别字。改一个文档里的 typo，提交一个 Pull Request，被合并了——恭喜你，你就是一个开源贡献者了。”

“这听起来很 trivial，但它帮你走通了整个流程：Fork → 改代码 → 提 PR → Code Review → 合并。流程走通了，以后做更大的贡献就有信心了。“

方式三：做自己的开源项目

“你写了一个小工具、一份学习笔记、一个课程项目的框架——把它开源放到 GitHub 上。写好 README，加一个开源协议（MIT 或 GPL）。”

“你不需要做一个很大的东西。你做的猜数字游戏、你的课堂笔记整理、你写的某个小脚本——这些对你来说很简单的东西，对下一届新生来说可能非常有价值。”

“这也是我们写《理工科计算机基础指北》的初衷——我们曾是新手，所以我们更懂新手；我们受过帮助，所以我们理应成为那个帮助别人的人。“

方式四：给开源项目贡献代码

“等你技术更成熟了，可以尝试给大的开源项目贡献代码。从标记为 ‘good first issue’ 的简单问题开始。”

“你给 PyTorch 提交了一个 bug fix，给 Linux 内核改了一行代码——这些都会永远记录在项目的贡献历史里。全世界任何人用这个软件，底层都有你的一行代码。”

“这种感觉，比考试考了 100 分爽多了。“

五、标志性开源项目（1 分钟）

“最后列几个你们以后一定会接触到的标志性开源项目。不用现在深入了解，知道它们存在就好。”

项目 说明 对你们的意义
Linux Kernel 全世界最重要的操作系统内核 理解 OS 的最佳教材
PyTorch / TensorFlow 深度学习框架 学 AI 必用
Hugging Face Transformers AI 模型库 几万个开源模型，直接调用
ROS 机器人操作系统 对机器人/自动驾驶感兴趣的方向
RISC-V 开源指令集架构 开源硬件的代表，理解计算机组成原理的好素材
Kubernetes 容器编排系统 云原生时代的基础设施
LLaMA / DeepSeek 开源大语言模型 AI 时代的核心基础设施

“这些项目的共同点是：它们都是由全世界的开发者协作完成的，源代码完全公开，任何人都可以参与。“

“你们以后学到的每一门课，几乎都能找到对应的开源项目去读、去用、去参与。”

收尾

“我花 10 分钟讲开源，是因为我觉得开源精神是 CS 学生最应该具备的底层精神之一。”

“计算机科学最幸运的一点是——它的全部知识、工具、社区，几乎都对所有人敞开。”

“一个历史系学生想自学前沿考古技术可能举步维艰。但一个计算机系学生，只要有网、有电脑，就可以直接触达世界顶级大学的教学视频、工业界的最佳实践、开源社区的最新代码。”

“你们被给予了人类历史上最开放的知识环境。请善用它。“

“好了，接下来看具体的 AI 工具——其中很多，也是开源的。”

“今天的开源部分补充资料：”

入门必看：

  • 如何给开源项目提交第一个 PR：搜”GitHub first pull request tutorial”

  • 开源协议入门：搜”MIT GPL Apache 区别”（简单了解即可，5 分钟看完）

  • 值得关注的开源社区：

  • GitHub Trending（github.com/trending）：每天看看什么项目在火

  • First Timers Only（firsttimersonly.com）：专门为开源新手准备的入门指南

  • Good First Issues（goodfirstissue.dev）：收集了各项目中标记为”新手友好”的 Issue

  • “今天回去做一件事：打开 GitHub Trending，看看今天最火的开源项目是什么。不一定要看懂代码，先感受一下开源世界有多大。”

第八块：资源推荐 + 收尾（10 分钟）

“好，今天讲了一大堆东西。我帮你们梳理一下，给你们推荐最精华的资源。”

“我只推荐每个类别最好的 1-2 个，不贪多。资源太多反而不知道看什么。”

入门视频课

资源 说明
CS50（哈佛大学） 全世界最好的计算机入门课，没有之一。B站搜”CS50”有中文字幕。强烈建议你们军训期间开始看第一课。
翁恺 C 语言（浙大） B站搜”翁恺 C 语言”，国内最好的 C 语言入门课。学校教 C 的时候可以配合着看。

书籍

资源 说明
《编码：隐匿在计算机软硬件背后的语言》 不是教你写代码的，而是用最通俗的方式告诉你计算机是怎么工作的。适合大一读，非常好读。
《C Primer Plus》 C 语言最经典的教材，厚但讲得清楚。
《大话数据结构》 数据结构入门，用大白话讲，比课本好懂。

练习平台

资源 说明
LeetCode 刷算法题的标准平台。从 Easy 开始做，大一不需要做太多，每周 2-3 道就行。
洛谷 国内算法竞赛平台，入门题目比 LeetCode 更友好。

工具

工具 说明
VS Code 编辑器，免费，必装
Git 版本管理，必装
Typora 或 VS Code 写 Markdown
Overleaf 写 LaTeX（不急，以后再说）
regex101.com 练正则表达式
Bitwarden 密码管理器

社区与信息源

资源 说明
GitHub 看别人的开源项目是最高效的学习方式之一
Stack Overflow 遇到 bug 第一个去搜的地方
Google 技术搜索首选
Hacker News（news.ycombinator.com） 技术新闻社区，了解行业动态

“这些资源我都会整理到《CS 新生生存手册》里面，今天讲座结束就发到群里。”

总结寄语

“今天讲了很多，从搜索技巧到 Markdown 到 Git 到命令行到正则表达式到密码管理……我知道信息量很大。”

“你们不需要今天就全部掌握。但至少要记住两件事：”

“第一，这些东西存在。 以后你遇到某个需求——比如需要写文档、需要管理代码版本、需要处理文本——你能想到’哦，有个工具叫 XX，我听过，去查查怎么用’。这就够了。”

“第二，动手比看书重要十倍。 今天回去做一件小事就够了：打开 VS Code，用 Markdown 写一篇今天的听课笔记，push 到你的 GitHub 上。就这一件事，做完你就比 90% 的同龄人领先了一步。”

留作业

“三个小任务：”

作业一： 用 Markdown 写一篇今天的听课笔记（不需要长，几行就行），push 到你 GitHub 上的仓库里。如果还没建仓库，今天建一个。”

作业二： 去 CS50 的 B站页面，看第一课。大概 1 个多小时。军训期间找时间看完。”

作业三： 给你的 GitHub 账号开两步验证。Settings → Password and authentication → Two-factor authentication。”

讲后群内运营

讲座结束后在群里发：

“今天的 PPT 和演示内容发群里了。重点回顾：”

  1. 搜索用 Google，搜英文，报错信息直接搜

  2. Markdown 从今天开始用起来

  3. Git 的基本流程：改 → add → commit → push

  4. 命令行不要怕，用到再查

  5. JSON/YAML 是数据交换格式，知道就行

  6. 密码不要提交到 Git，开两步验证

  7. “下周四讲 AI，是整个系列最重要的一次。想听的同学别错过。另外，军训辛苦了，还有两周，加油 💪”
